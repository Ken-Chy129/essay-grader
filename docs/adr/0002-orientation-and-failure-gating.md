# 0002 — 方向校正交给工程+人,LLM 不碰方向;上传与评分解耦

- 状态:已接受 (Accepted),取代早期"LLM 判方向"的做法
- 日期:2026-06-30

## 背景 (Context)

手机拍的手写作文常常旋转躺倒。最初让 LLM "判断该转多少度"再转正,实测**很不可靠**:
方向判错时,模型**不会拒绝,反而会凭空编造一整篇通顺却完全不存在的作文**,并照样得到一个
看似合理的分数(实测把一篇绿笔作文"批改"成了关于"保持本心/信息爆炸"的范文,30/60)。
这是批改产品最严重的信任崩塌——给一篇根本不存在的作文打分。

根因:**方向是确定性的图像属性,却交给了概率性的 LLM 去猜**;而 LLM 在"看不懂"时倾向编造而非认错。

## 决策 (Decision)

1. **LLM 完全不参与方向判断。** 它只接收"已确认正立"的图。
2. **方向校正 = 工程 + 人:**
   - 工程:上传时读 **EXIF 朝向**自动转正(纯 Go,无依赖)——真手机照片大多一步到位;
   - 人:界面每张缩略图右上角有 **↻ 旋转**,老师扫一眼把剩下歪的转正(纯图像旋转)。
3. **上传与评分解耦:** 上传只做"存图 + EXIF 转正",状态=`uploaded`,**不批改**;
   老师确认方向后点 **「开始评分」**,才进入转写+评分。
4. **兜底:** 万一转写仍自述"方向不对"或拒绝,判 `error`、**绝不送去评分**,提示老师旋转重试。

## 后果 (Consequences)

- 从根上消除"方向错→胡编范文→错误分数":LLM 只看人确认过的正图。
- 方向校正确定、可见、可控(所见即所得),不依赖概率。
- 上传快(无模型调用、无等待);评分是一次显式、可预期的批量动作。
- 代价:EXIF 缺失(如截图、个别机型)时需老师手动转一下;HEIC 等 Go 无法解码的格式需另转。

## 被否决的备选 (Alternatives)

- **LLM 判方向(0/90/180/270)**:实测产生 180° 误判并诱发整篇幻觉,已证伪。否决。
- **本地 OCR 引擎方向分类(Tesseract OSD / PaddleOCR)**:全自动但引入重依赖,且对手写方向准确率一般。本期不引。
- **纯图像算法判 180°**:手写"正立 vs 颠倒"无可靠的非 ML 解法,理论上做不到。
